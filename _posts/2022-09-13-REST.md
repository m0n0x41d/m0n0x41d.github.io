# Собственно, not in Peace

REST — это очередная штука, про которую должен знать и понимать каждый уважающий себя ~~гик~~ инженер. За аббревиатурой стоит Representational State Transfer. Как мне кажется, чтобы быстрее запомнить некоторый IT терми, очень важно развернуть его сокращение и понять название, вместо того чтобы мозг автоматически формировал в себе какие то непонятные ассоциации, хотя они тоже могут быть полезны для запоминаня :)

Итак, ээм... "Репрезентативная" передача состояния? Короче говоря, суть в передаче состояния, это понятно. А под репрезентативным будем предполагать, что это состояние должно передоваться некоторым *понятным*, самоописывающим способом. Понятно? Нет, ничего не понятно. Давайте разбираться.


# API, тащемта.

Я не писал отдельный пост про API, потому что вы ведь итак знаете что такое API, да?

API — Application Programming Interface, который можно так же как мы только что пытаться переводить и выносить себе мозг, но думаю с вас на сегодня хватит. Апи — это набор определений интерфейсов и протоколов, которые служат для интеграции приложений (программного обеспечения) для построения некой системы.

Иногда Апи описывают как "контракт" между поставщиком информации и её потребителем, который определяет "контент" требуемый каждой из сторон для выполнения своих "обязанностей" по этому "контракту".

Короче говоря, если кто-то хочет поработать с N системой для получения какой то информации, или попросить эту систему выполнить некоторую "функцию", API поможет сообщить этой N системе чего от неё вообще хотят, в понятной для неё форме.

Понятнее стало? Это слой абстракции.


# Итак, REST

Сам по себе REST — это *архитектурный стиль*, представленный набором правил для организации *структуры* бэкенд приложения. Считается, что построенное следуя REST приложение можно легко масштабировать. Хотя, это верно скорее в разрезе компетенции разработчика, так как в слепую следуя любому паттерну и стилю далеко не уедешь.

В контексте веб-приложений используется термин RESTful, если такое приложение *не нарушает ограничений, которые накладывает REST стиль*.

## Бегом по ограничениям REST

Итак, когда основные свойства REST Архитектуры:
- Производительность
- Масштабируемость

нам интереснее рассмотреть вкратце ограничения, накладываемые REST:

- Клиент-серверная модель;
- Отсутствие состояния; Никакой информации о состоянии клиента на сервере не храним. Состояние сессии хранится у *клиента*.
- Поддержка кэширования;
- Единообразие интерфейса; Это фундаментальное требование, возможно самое важное. Основная идея тут в том что каждый ресурс должен идентифицироваться в запросе (URI);
- Слои; Клиент не может точно определить, да и не должен, с кем он общается сейчас — напрямую с сервером, или с проксей? может вообще CDN? Это не должно никак влиять на работу приложения;
- Код "по требованию"; Дополнительное, *не обязательное* ограничение. В общем это расширение, позволяющее прокачать клиента путем загрузки некоторого кода с сервера.

## RESTful API, HTTP

Официального стандартра RESTful API попросту не существует, как минимум потому что REST — не протокол, а люди любят официально стандартизировать, похоже, только протоколоы. В этом есть рациональное зерно. Что тут стандартизировать? Вот тебе ограничения архитектруного стиля, садись и прогай, будь мужиком!

Про прикладной протокол [HTTP мы говорили тут](/2022/05/22/HTTP.html). Первое ограничение помните ещё? Клиент-серверная модель. То-то и оно.

Продолжим в контексте сегодняшней темы.

Мы помним о том что HTTP определет набор методов, которые отображают суть того, чего мы хотим от ресурса к которому обращаемся. Их называеют ещё HTTP Verb, глаголы... Ну, в общем, призыв к действию :)

Наиболее часто использумые "призывы к действию":

- GET: Запрашиваем представление ресурса;
- HEAD: То же самое, что GET, только получаем ответ *без body*;
- POST: Отправляем некоторые данные ресурсу, часто вызывая этим изменения в состоянии хранимом на сервере, или другие сайд-эффекты;
- PUT: Полностью заменеяем текущее представление указанного ресурса передаваемой в этом запросе "полезной нагрузкой";
- DELETE: Удаляем ресурс.
- PATCH: Частично модифицируем ресурс некоторым образом.
